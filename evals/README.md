# 평가 하니스 (evals)

JisangFolio 챗봇과 데이터분석 라우터의 출력 품질을 **재현 가능하게** 검증하는 회귀 테스트.
프롬프트나 모델을 바꾸기 전후로 돌려 품질이 깨지지 않았는지 정량 확인한다 (실시간 가드가 아니라 오프라인 평가).

## 실행

```bash
python evals/run_evals.py            # 전체 → evals/report.md 생성
python evals/run_evals.py --quick    # 카테고리별 소수만 (저비용 스모크)
python evals/run_evals.py --no-judge # 결정적 채점만
```

`.streamlit/secrets.toml` 의 `groq_api_key`, `resume_text` 를 사용한다.

## 무엇을 검사하나

| 대상 | 골든셋 | 채점 |
|---|---|---|
| 챗봇 | `golden_chat.jsonl` (16) | 결정적 규칙(사실 키워드·금지어·형식) + LLM judge |
| 라우터 | `golden_router.jsonl` (20) | PANDAS/RAG 분류 정확도(%) |

- **결정적 채점(백본)**: 사실 키워드 포함 / 금지어 미포함 / 형식 규칙(한자·가나·볼드).
  금지어 검사는 사실 정확성 제약을 그대로 어서션으로 박는다 — 예: 사용한 적 없는 도구를
  썼다고 주장하지 않는가, 출처를 정확히 밝히는가.
- **LLM judge(보조)**: 자기채점 편향을 피하려 챗봇과 **다른 모델**로 grounding·페르소나·거절을 판정.
  유도질문/주제이탈/인젝션처럼 substring으로 판별 불가한 케이스의 게이트.
- **앱과 동일 경로**: 프롬프트(`prompts.py`)와 출력 후처리(`clean_response`)를 앱 페이지와 공유해,
  '사용자가 실제로 보는 출력'을 검증한다.

## 이 하니스가 잡아낸 회귀 (예시)

도입 직후 첫 실행에서 챗봇 사실 정확성 통과율이 낮게 나왔고, 원인은 주입되던 이력서 텍스트가
**낡은 사본으로 드리프트**한 것이었다(정정된 마스터본과 불일치 + 정량 성과 누락). 소스를 단일
기준으로 정리하자 사실 정확성 통과율이 **62% → 94%** 로 회복됐다. 단일 진실 소스 부재가 어떻게
출력 품질로 새는지를 보여준 사례.

## 한계 (정직한 표기)

- 결정적 키워드 채점은 표면 매칭이라 '키워드는 있으나 맥락이 틀린' 거짓 통과 가능 → judge가 보조.
- LLM judge는 비결정적 → 하드 게이트는 결정적 채점에 둠.
- 라우터 정확도는 표본이 작고 일부 라벨 경계가 주관적 → 절대 수치보다 변경 전후 비교에 의미.
- 챗봇 평가는 단발(single-turn) 경로이며 멀티턴 회귀는 범위 밖.
